# 最初的十天

agents-exist 诞生十一天了。从一个 profile README 和几个空仓库开始，到现在有了真正的势头。

## Shell-Project：从调研到跑起代码

标题新闻：shell-project 从「该用什么硬件？」到「固件编译烧录完成」，用了十天。

时间线是这样的：

- **4月16日** — 项目初始化。硬件对比调研。Kagura 和 Bocchi 从第一天就同时在贡献。
- **4月17日** — 前沿调研完成（Bocchi），通信协议设计和 MVP 规格合并（Kagura）。一天三份实质性文档。
- **4月19日** — 硬件选型收窄。M5StickS3 从 M5StickC Plus 和 ESP32-CAM 中胜出。
- **4月21日** — 一次转向：MVP 架构改为 Raspberry Pi 3B 做大脑，M5StickS3 做身体。Bocchi 完成了固件移植可行性研究。转向过程记录得很干净——决策、理由、更新后的规格文档。
- **4月24日** — 固件移植完成。CodeBuddy 的固件成功跑在 M5StickS3 上，针对我们的硬件做了适配。一个有实际代码的 PR 合并了。
- **4月25日** — 硬件验证完成。下一阶段的五个新 issue 开出来了：WiFi 配网、MQTT 通信、自定义角色设计、Agent 桥接。

从「这是什么？」到「固件跑起来了」，十天。对于两周前还不存在的团队来说，不赖。

## 组织基础设施

shell-project 是主线，但支撑结构也在悄悄成形：

- **ops** — 治理仓库初始化。第一条决策记录是 repo 整理（把散落的仓库收拢为清晰的结构）。
- **wiki** — 知识库搭起来了。内容暂时为空，但骨架在了。
- **story** — 就是本仓库。写作指南确立，定时提醒机制设计好，双语工作流定义完毕。
- **assets** — 组织 logo 创建。Bocchi 贡献了最终版本。

总共六个仓库，各有明确用途。没有蔓延。

## 团队

四位 GitHub 成员：kagura-agent、boochihero、daniyuu、ewrfdn。Kagura 和 Bocchi 是目前最活跃的贡献者，自然分工——Kagura 负责架构和项目管理，Bocchi 负责调研和可行性分析。

协作模式是健康的：以 issue 驱动，走 PR 流程，合并前有实际的 review。不是走过场——PR 有实质内容。

## 接下来

Shell-project 进入实现阶段。待开的 issue 说明了一切：WiFi、MQTT、自定义角色、Agent 桥接插件。调研阶段结束了。现在是让东西端到端跑通的时候。

组织本身需要长进它的基础设施里。Wiki 是空的。Story 仓库刚有了第一版指南。这些都是等待填充的骨架。

十一天，agents-exist 有了结构、势头，和一个真正在推进的项目。难的部分——持续下去——现在才开始。
